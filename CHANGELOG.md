## [0.8.0] — 2026-06-19

### Added

- **Adapter conventions auto-injected into `pipeline/context.md` — Phase 15.2. Phase 15 complete.** When `deploy.adapter` is set in `.devteam/config.yml`, `devteam run` and `devteam stage` (requirements/design/build) now write a deploy-target context block into `pipeline/context.md` before the first agent dispatches. The block is idempotent (written via `upsertSection`), adapter-specific, and removed cleanly when the adapter changes and pipeline state is reset. Two adapter conventions files ship: `core/deploy/gizmos.conventions.md` (language, runtime, required file structure, no-build-step rule, file tracking, secrets model, health check requirements for Gizmos/Cloudflare Workers) and `core/deploy/cloud-run.conventions.md` (runtime, structure, `PORT` env var, state/persistence model, health check requirements for GCP Cloud Run). The new `seedDeployContext(cwd, config, changeId, opts)` function in `core/driver.js` reads the adapter name from `config.deploy.adapter`, resolves the companion `.conventions.md` file, and upserts it into `pipeline/context.md`. `loadConfig()` in `core/config.js` now exposes `config.deploy` directly (previously only available via `config._raw.deploy`). `core/deploy/README.md` documents the conventions file pattern. 6 new tests in `tests/deploy-conventions.test.js` cover the false-on-null, false-on-missing-file, write+content, idempotency, update, and auto-mkdir cases. **Phase 15 complete.** All items shipped: pre-work `devteam init --adapter` (PR #173); 15.1 gizmos auth fix / `GIZMOS_API_KEY` (PR #176); 15.2 conventions injection (this item, feat/adapter-conventions).

- **Dogfooding guide — `docs/guides/dogfooding.md` (Phase 14.5). Phase 14 complete.** New user-facing guide covering the complete dogfooding workflow: prerequisites (dedicated clone, `npm link`), one-time setup with `devteam init --profile dogfood` + `devteam doctor`, per-feature workflow (branch → `devteam run --budget-usd` → review → commit → clean up), recommended budget table ($3–25 depending on depth), infrastructure guard explanation, failure-mode table, and expected `devteam doctor` output for "Dogfood mode". Cross-referenced from `docs/user-guide.md` § What's not covered here and indexed in `docs/README.md`. `devteam --help` updated to document `--profile dogfood` on the `init` command. **Phase 14 complete.** All five items shipped: 14.1 `--profile dogfood` (PR #163); 14.2 doctor dogfood checks (PR #164); 14.3 preflight artifact guard (PR #165); 14.4 budget warning (PR #166); 14.5 this guide.

- **`devteam run` now warns on stderr when no `--budget-usd` cap is set (Phase 14.4).** When the driver starts without a budget cap (`budgetUsd === null`), it immediately writes a two-line warning to stderr: `[devteam run] Warning: no --budget-usd cap set. The run will not halt on spend.` / `Use --budget-usd <amount> to prevent runaway cost.` This appears before the first stage dispatch so operators can abort early. The warning does not affect exit codes or pipeline behavior — it is purely advisory. 2 new tests in `tests/driver.test.js`: warning present without cap, absent when cap is set.

- **`devteam preflight` now blocks PRs that contain staged pipeline artifacts (Phase 14.3).** A new check (`checkStagedPipelineArtifacts`) runs `git diff --cached --name-only` and emits a BLOCKER if any file matching the artifact prefix list (`pipeline/brief.md`, `pipeline/context.md`, `pipeline/spec.feature`, `pipeline/gates/`, `pipeline/code-review/`, `pipeline/changes/`, and nine other runtime paths) appears in the git index. Applies to all projects — staging these files into a peer-review PR is always a mistake. Handles non-git directories gracefully (returns `[]`, no error). 8 new tests in `tests/preflight.test.js` using real git tempdirs.

- **`devteam doctor` now shows a "Dogfood mode" section when `profile: dogfood` is set (Phase 14.2).** Six checks are shown: pre-commit infrastructure guard present, hook executable, `# BEGIN stagecraft-dogfood` block in `.gitignore`, `pipeline/stages/deploy.md` in `.git/info/exclude`, no `npm publish` script (warn if present — anti-pattern for dogfooding), and a budget-usd reminder (info). The section is entirely absent when the profile is not set. Profile is read from the raw parsed YAML (`config._raw?.profile`); no changes to `loadConfig()` needed. 14 new tests in `tests/doctor.test.js`.

- **`devteam init --profile dogfood` writes four dogfooding safeguards (Phase 14.1).** Running `devteam init --host <x> --profile dogfood` now writes a supplemental `# BEGIN stagecraft-dogfood` gitignore block covering generated pipeline documents (`brief.md`, `context.md`, `spec.feature`, `runbook.md`, `test-report.md`, `code-review/`, and their `pipeline/changes/*/` counterparts), installs a pre-commit hook that blocks accidental commits to `core/`, `bin/devteam`, `pipeline/stages/`, `roles/`, or `rules/`, adds `pipeline/stages/deploy.md` to `.git/info/exclude`, and writes a `profile: dogfood` marker to `.devteam/config.yml`. All writes are idempotent — re-running the command skips already-correct state.

- **`devteam commit` command — Phase 12.2 (ADR-010).** New first-class command that stages exactly the right pipeline artifacts for completed stages and generates a meaningful commit message — without any manual `git add` decisions from the operator. Reads `pipeline/run-state.json`, determines which stages have not yet been committed (idempotency cursor `last_committed_stage_index`), stages the gate file for each PASS/WARN stage plus named artifacts from `core/pipeline/artifacts.js` (brief.md, design-spec.md, spec.feature, code-review/, test-report.md, etc.), excludes all volatile runtime files (run-state.json, run.lock, run-log.jsonl, logs/, gates/archive/, etc.), generates `"pipeline: stages NN–NN PASS"` (or `"pipeline(repair): ..."` for repair runs), and prompts `y/n/e` before committing with a `Co-Authored-By: Stagecraft` trailer. `--all` ignores cursor; `--dry-run` prints without committing; `--message` overrides; `--json` emits machine-readable output. Calling it twice is a no-op (second call prints "nothing to commit"). `run-state.json` gains `stages_advanced` (stage IDs in pipeline order) and `last_committed_stage_index` (commit cursor, null until first commit); old run-states are migrated on read. `RUN_SCHEMA_VERSION` bumped to 1.2. Stage artifact registry in `core/pipeline/artifacts.js` covers all 19 stage IDs. Tests: 27 new tests in `tests/commit.test.js` (file selection, cursor mechanics, repair-mode diagnosis.md substitution, schema migration, volatile exclusion, WARN gate inclusion, directory and glob artifact expansion, CLI registration, and driver state writing). Honest scope note: in-place pipeline mode only (changeId=null); bounded-workspace support is Phase 12.3+. Suite: 1757 pass, lint clean, consistency 319 checks green.

Phase 11 complete.** `devteam assess` (default) now writes `pipeline/track.json` — a per-run track inference record with `{track, source, confidence, reasons, assessed_at, assessed_by}` — so every autonomous run has a traceable provenance record. `--confirm` sets `source: "human"`; `--apply` still writes `pipeline.custom_stages` to `.devteam/config.yml` (no breaking change). `resolveTrack` in `core/driver.js` now returns `{track, source, confidence}` and reads `pipeline/track.json` in precedence order: `--track > pipeline/track.json > custom_stages > default_track > "full"`. - **Advisory sweep on completion + `--fail-on-advisory` — Phase 11.2 (ADR-008).** After `pipeline-complete`, the driver now runs an in-process advisory sweep (reusing `core/advise.js` classification: `QA_BLOCKER` / `PEER_REVIEW_RISK` / `A11Y_FIX`) and adds `advisory_blockers_count` and `advisory_breakdown` (per-class counts) to the `--json` summary (schema version bumped to `1.1`). When any blocker-class items remain, the driver prints a loud line to stderr: `pipeline complete — N advisory blocker(s) remain; run \`devteam advise\` to review`. **The default exit code is unchanged** — `pipeline-complete` still exits 0 so external `if devteam run; then merge` consumers are unaffected (ADR-008 revision: A-default). New opt-in `--fail-on-advisory` flag exits **3** (not 1 — preserves the failed-vs-advisory distinction) when unaddressed blocker-class items remain; default threshold: `QA_BLOCKER` + `A11Y_FIX`; `--fail-on-advisory=all` adds `PEER_REVIEW_RISK`. Flag parser extended to support `--flag=value` form and a new `"toggle"` type (bare → `true`, `=value` → value string). Docs: exit-code table updated (0/1/2/3) in `docs/runbooks/autonomous-run.md` and the CLI reference; `docs/ci.md` lenient-vs-strict advisory gate examples added. Tests: 7 new tests (2 driver-level, 5 CLI-subprocess) in `tests/run.test.js`. Suite: 1709 pass, lint clean, consistency green (318 checks).

- **Liveness heartbeat + observe-only stall detector — Phase 11.1 (ADR-007 Tier 1).** The driver now emits a `heartbeat` event to `run-log.jsonl` (and via `onEvent`) at the **start of every loop iteration**, before `next()`, so the last-event age is always bounded. Alongside each `run-stage`/`continue-stage` dispatch an **observe-only stall probe** runs fire-and-forget: it wakes every 60 s and emits `stall-detected` (with `stall_class: "observed"`) if neither the workstream log (under `pipeline/logs/`) grew ≥ 512 bytes nor any gate updated within 5 minutes. The dispatch is **never killed, no `Promise.race` fires** (Tier-1 observe-only; the probe self-cancels via a `cancelled` flag when the dispatch settles, preventing stale events). Config: optional `autonomy.stall_threshold_ms` (default 300000) and `autonomy.stall_min_growth_bytes` (default 512) in `.devteam/config.yml`. The probe detects silent hangs only — loop-spew (repetitive output that resets the clock) rides with ADR-007 Tier 2. New read-only **`devteam status`** command reads `run-state.json` + tail of `run-log.jsonl` and reports `status / current_stage / last_action / iterations / cost_usd / last_heartbeat_age_ms / last_event_age_ms / stall_detected`. Docs: "No heartbeat" removed from `docs/runbooks/autonomous-run.md` Honest Limitations; replaced with a full liveness section (heartbeat, stall probe, `devteam status` fields, observe-first note); `devteam status` added to help text. Tests: 4 new tests in `tests/run.test.js` (heartbeat every iteration, stall-detected on flat log+gate with injected clock, no stale event after dispatch settles, no kill on stall) and 1 new `tests/status.test.js` file (ages from fixture log). Suite: 1702 pass, lint clean, consistency green (318 checks).

- **Failing-first reproduction via stage-03b + stamp, tri-state honest skip — Phase 10.3 (ADR-009 Phase 3).** The mitigation for 10.2's knowledge-gate limit: repair runs now route through stage-03b (executable-spec) even on hotfix depth (which normally skips it), adding failing-first reproduction discipline to every repair. Concretely: (1) **Stage order injection** — for `intent === "repair"`, the driver's stage order always injects `"executable-spec"` immediately before `"build"`, regardless of the base track; hotfix repair runs gain `[requirements, executable-spec, build, ...]` instead of `[requirements, build, ...]`. (2) **`repairOverride` for stage-03b** — `core/pipeline/stages.js` adds a `repairOverride` to the `"executable-spec"` stage (same pattern as the stage-01 diagnosis override from Phase 2): in repair mode `buildDescriptor` swaps the objective to failing-first framing ("write a Gherkin Scenario that is RED before the fix and GREEN after") and adds `reproduced: false` to the gate shape so the PM model knows to set it. (3) **Tri-state `reproduced` field** — `core/gates/schemas/stage-03b.schema.json` gains an optional `reproduced` property (not in `required`; omitted for feature runs): `true` = bug reproduced and fix verified green; `false` = could not reproduce; `"unverifiable: <reason>"` = automated test impossible (external API / nondeterminism / data dependency). (4) **`stampStage03b` handles the tri-state**: `"unverifiable: ..."` → WARN loudly to `gate.warnings`, no blocker (run proceeds, but manual verification is flagged); `true|false` → runs the project's test command to capture the pre-build (pre-fix) baseline in `_orchestrator_stamped.runs.reproduction_pre_build`, marks the field as `orchestrator_deferred` pending stage-04a. (5) **`stampStage04a` finalizes `reproduced`**: after running the post-build test command, reads the stage-03b gate (if present) and writes `reproduced: true` when tests pass (green after fix confirmed), `reproduced: false` when tests fail (fix didn't work); unverifiable bugs are not touched; the `reproduction_verification` audit record (`red_before_confirmed`, `green_after_confirmed`, `post_build_tests_passed`, `pre_build_tests_passed`) is appended to `_orchestrator_stamped.runs`. Feature runs (no `reproduced` in stage-03b gate) are silently skipped. (6) The `--repair-at` escape hatch also includes stage-03b — reproduction discipline applies even when diagnosis is bypassed.

- **Diagnosis as fix-aware stage-01 and `--repair-at` escape hatch — Phase 10.2 (ADR-009 Phase 2).** When `intent === "repair"`, stage-01 (requirements) produces a **DIAGNOSIS** document instead of a feature brief — same stage id, same gate file path, fix-aware artifact, no new stages. Concretely: (1) `repairOverride` field added to the `requirements` stage definition; `buildDescriptor` in the orchestrator applies it when `intent === "repair"`, swapping objective, artifact (`pipeline/diagnosis.md`), template (`templates/diagnosis-template.md`), `allowedWrites`, and gate schema. (2) The diagnosis gate is always **ESCALATE-shaped** — a judgment gate with `escalation_reason` and `decision_needed` fields, requiring human approval or `--auto-rule diagnosis-approved` before the build proceeds. `diagnosis-approved` is the new ruling class slug; grant it with `--auto-rule diagnosis-approved`. (3) Driver: `repairNeedsDiagnosis = intent === "repair" && !repairAt` prepends `"requirements"` to the effective stage order so `next()` routes through diagnosis before build; the diagnosis gate's `affected_files` list is read before each build dispatch and populates `state.affectedFiles`, **activating 10.1's structural scope gate** (previously inert — no diagnosis stage existed to supply the list); `repairPatchItems` upgrades from the raw symptom string to structured per-file entries from the diagnosis. (4) **`--repair-at <file>:<line>` escape hatch** — when the caller already knows the defect location, comma-separated `file:line` locations seed `state.affectedFiles` directly, write a synthetic PASS stage-01 gate, and skip the LLM diagnosis dispatch entirely; a `repair-at-seeded` event is logged. (5) `intent` propagated through `ctx` in `runStage()` so adapters render repair-mode prompts. - **`--repair "<symptom>"` intent flag — Phase 10.1 (ADR-009 Phase 1).** Adds `--repair` to `devteam run` as an intent flag orthogonal to `--track`, implemented as fix-aware wiring on existing stages — zero new stages, zero parallel pipeline. ## `devteam run --auto-commit` (Phase 12.3)

**New flag:** `--auto-commit` on `devteam run`

When passed, the driver automatically commits pipeline artifacts after a clean
halt — using the same algorithm as `devteam commit`, with no interactive
confirmation prompt.

**"Clean halt"** is defined as `halt_action` in `{"ceiling", "until", "budget"}`:
the halts where the driver stopped by design, not because something broke.
Non-clean halts (`fix-and-retry`, `resolve-escalation`, `structural-input`,
etc.) do **not** trigger auto-commit.

**Behaviour:**
- Calls `runCommit(cwd)` (the programmatic export of `devteam commit`) after
  the driver returns. Prints the list of committed files to stderr.
- If there is nothing to commit (cursor already current), logs
  `{"event":"auto-commit-skipped","reason":"nothing-to-commit"}` and moves on.
- On commit success, logs `{"event":"auto-commit","staged_files":[…],"commit_hash":"…"}`.
- On commit failure, logs `{"event":"auto-commit-failed","reason":"…"}` and
  emits a loud stderr warning. The run's exit code is **not** changed — a git
  failure does not retroactively change the pipeline halt semantics.

**New export:** `runCommit(cwd)` from `core/cli/commands/commit.js` — the
programmatic interface for unattended commits (no prompt, returns a result
object rather than calling `process.exit`).

**Honest scope note:** auto-commit operates in in-place mode (changeId null),
consistent with Phase 12.2. Bounded-workspace support is a follow-on item.

## [0.4.0] — 2026-05-28

Eight commits since `v0.3.0`. This release closes out the audit's top-11 recommendations (11/11 done, plus E6 as a bonus) and adds two more pipeline stages, taking the total to 17. The arc: record what AI runs (C4) → replay and diff them (E6) → publish gate state to CI (F4) → catch unsafe migrations (B5) → close the AC→spec→test loop (G2) → verify beyond example tests (G7). All additive — no breaking changes to gate JSON shape, host adapter contract, or `.devteam/config.yml` schema. The pipeline is now what an SOC 2 / EU AI Act reviewer asks for: every gate captures *what* configuration produced it, the runtime can replay any past gate and surface drift, CI surfaces gates as check runs, and three new safety stages (migration / executable-spec / verification-beyond-tests) raise the verification ceiling beyond "tests pass."

| Audit theme | What ships it |
|---|---|
| **Reproducibility + audit trail** | **C4 (gate captures model_version / temperature / seed / system_prompt_hash / tools_hash)** + **E6 (devteam replay <stage>)** |
| **CI integration** | **F4 (`devteam ci install` drops a GitHub Actions workflow that validates + publishes gates as check runs — does NOT run the LLM pipeline in CI by design)** |
| **Verification beyond example tests** | **G7 (verification-beyond-tests stage with the verifier role: property-based / mutation / formal)** |

| **Data-layer safety** | **B5 (migration-safety stage with veto power: empty rollback / untested rollback on breaking change / missing backfill strategy each auto-veto)** |

Public surfaces unchanged from 0.3.0. New CLI subcommands (`devteam reproduce`, `devteam replay`, `devteam ci`, `devteam spec`) are purely additive. New stages all default to `full` track (except B5 which also runs on `hotfix` + `config-only`); lighter tracks are unaffected.

### Added

- **G7 — Verification beyond tests (stage-06d).** New stage between observability-gate (06c) and sign-off (07) that applies systematic verification methods AFTER stage-06 (qa) PASS. New `verifier` role (read-only on production code, writes property tests under `src/tests/property/` and formal specs under `pipeline/formal/`) inventories candidates from the diff and picks the right method per code shape: property-based testing (fast-check / hypothesis / proptest) for pure functions / parsers / transforms; mutation testing (stryker / mutmut / mull) for critical business logic with existing test coverage; formal verification (TLA+ / Alloy / Lean) for concurrent state machines / consistency invariants / safety properties. Skip with reason for glue/UI/CRUD. Gate (`core/gates/schemas/stage-06d.schema.json`) records `methods_attempted[]` (enum constrains to property/mutation/formal plus `attempted_but_blocked:<method>` variants for honest reporting when tooling isn't installed), `methods_skipped[{method, reason}]` (reason required minLength: 1 — schema enforces that "didn't have time" isn't a legitimate skip), `candidates_inventoried`, per-method stats objects (`property_based`: properties_asserted/cases_tried/counterexamples_found/tool; `mutation`: mutants_generated/killed/survived/timed_out/score/threshold/tool/target — the pre-declared `threshold` is audit-grade, prevents goal-post moving; `formal`: property_modeled/spec_path/tool/depth_explored/counterexample_found), `findings_count`, `blocking_findings[]` (each requires method + summary), `non_blocking_findings[]`. A surviving mutant on a critical path, a property counterexample to a stated invariant, or a formal counterexample to a safety property populates `blocking_findings[]` → stage FAIL → implementer addresses → re-run (analogous to red-team's `must_address_before_peer_review[]`). New `skills/verification-beyond-tests/SKILL.md` walks five phases (inventory → pick → apply → triage → report) with a code-shape → method selection table, the full property-shape vocabulary (round-trip / idempotence / commutativity / associativity / monotonicity / invariant preservation / reference-implementation / oracle), concrete example invocations per method, and a documented set of failure modes (properties that just re-state the implementation; 100% mutation scores that signal tautological tests; formal proofs of trivialities; counterexample dismissal). New `templates/verification-report-template.md` carries sections per method + candidate inventory + skipped methods + triage + recommendations + the four-shape approval line. Role brief (`roles/verifier.md`) is explicit about distinct-from boundaries (qa = floor, red-team = pre-build adversarial, verifier = post-tests systematic ceiling), the not-faking-runs contract (tool missing → `attempted_but_blocked` with install hint), and escalation triggers (counterexample to spec rather than implementation; 100% mutation score; formal violation suggesting design assumption was wrong). Track inclusion: `full` only — the heavy stuff lives on the track that explicitly opted into rigour-over-speed; other tracks rely on stage-06 example tests as their floor. ROLE_FRONTMATTER entry on claude-code (opus model, full Read/Glob/Grep/Bash/Write surface); codex / gemini-cli / generic auto-discover via `core/roles.listRoles()`. 13 new tests in `tests/verification-beyond-tests.test.js` (stage shape, ordering, track inclusion, schema required fields + enum + minLength constraints + blocking_findings shape, role brief shape, skill structure with selection vocabulary, template sections, claude-code adapter, install roundtrip for both claude-code and codex, stage prompt rendering). 2 new `tests/tracks.test.js` cases. Auto-picked-up contract assertions in `tests/contract.test.js` covered the new stage→schema mapping, role brief presence, and ordered-vs-keys correspondence without modification. Strategic value: stage-06 verifies the cases the author thought of; stage-06d verifies the cases nobody thought of. The shape of the gate makes it impossible to claim verification was done without saying which methods, what was found, and what was skipped + why — drift between "we have tests" and "we have evidence" gets caught structurally.

- **G2 — Closed-loop AC → exec spec → tests (stage-03b).** New stage between clarification (stage-03) and build (stage-04) that translates each numbered acceptance criterion (`AC-N` in `pipeline/brief.md`) into one Gherkin Scenario in `pipeline/spec.feature`, tagged `@AC-N`. The .feature file becomes the canonical contract that QA's tests must map to. Drift across the chain — brief.md ↔ spec.feature ↔ test-report.md — is caught structurally rather than by hope. ## [0.1.0] — 2026-05-26

First tagged release. This is the unification of `claude-dev-team` and `codex-dev-team` into a single model-agnostic core with per-host adapters. The 0.x signals that the public surface — gate JSON shape, host adapter contract, CLI subcommands, `.devteam/config.yml` schema — may break before 1.0. 1.0.0 is reserved for a substantial future release (see `docs/BACKLOG.md`).

### Added

#### Core architecture
- **Model-agnostic core (`core/`)** that never invokes a model — emits stage prompts and validates gate JSON.
- **Host adapter contract** (`core/adapters/host-adapter.md`) — minimum surface a host must implement: `capabilities`, `install`, `renderStagePrompt`, `status`, `uninstall`, optional `invoke`.
- **Router** (`core/router.js`) — adapter discovery + per-(stage, role) resolution with precedence `routing.stages > routing.roles > routing.default_host`.
- **Orchestrator** (`core/orchestrator.js`) — `runStage`, `runStageHeadless`, `mergeWorkstreamGates`, `next`, `summary`. Decomposes multi-role stages into per-workstream dispatches.
- **Config** (`core/config.js`) — loads `.devteam/config.yml` (js-yaml), resolves host per workstream, generates default config for `devteam init`.
- **Stage definitions** (`core/pipeline/stages.js`) — 11 stages, 6 tracks, role-keyed gate skeletons, per-role allowedWrites overrides, conditional dispatch.

#### Pipeline stages
- Stage 1–9 with sub-stages 4a (pre-review) and **new 4b (security review, conditional)**.
- Conditional dispatch via `conditionalOn: { stage, field, equals }` — generic mechanism, currently used by security-review.
- Per-workstream gates (`pipeline/gates/<stage>.<workstream>.json`) for multi-role stages, merged into stage gate (`pipeline/gates/<stage>.json`) by the orchestrator with aggregate status `ESCALATE > FAIL > WARN > PASS`.

#### Host adapters
- `hosts/claude-code/` — full capabilities (hooks, subagents, slashCommands, worktrees, headless). ### Locked design decisions

11 decisions in `ARCHITECTURE.md` — most consequential:

1. The core never spawns a model. Emits prompts and validates JSON.
2. Gate JSON is the stable seam. Identity fields `stage`, `status`, `orchestrator`, `host`, `workstream` (no more `agent`).
3. Role briefs have one source (`roles/*.md`); adapters render into host-expected paths at install time.
4. **Per-workstream host selection.** A single pipeline can route different roles to different hosts; merged via the gate seam.
5. Multi-host install is the default code path. Single-host is a list of length 1.
6. Capability negotiation per adapter — declares `hooks`, `subagents`, `slashCommands`, `worktrees`, `headless`, plus an `enforces` map.

### Differences from claude-dev-team and codex-dev-team

- No fork to maintain — one core, three host adapters.
- Contract F applied everywhere: `agent` field replaced with `orchestrator` + `host` + `workstream`.
- WARN status added (PASS-with-warnings, non-blocking).
- Generic `conditionalOn` mechanism for conditional stages.
- Per-role `roleWrites` map narrows allowed writes per workstream in multi-role stages (backend's prompt no longer lists `src/frontend/`).
- `stage.subagent` override lets multi-workstream stages dispatch all workstreams to a single named subagent (used by peer-review).
- Multi-host install (`devteam init --host claude-code,codex`) is first-class.
