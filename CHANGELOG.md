## [Unreleased]

Post-0.4.0 work, accumulated since `v0.4.0` (2026-05-28). Three themes:
(1) tightening the build loop so adversarial findings feed back into the
same stage that produced them (red-team blocker auto-injection +
`--patch`, QA-within-build); (2) operational ergonomics for unattended
runs (parallel workstream dispatch, `--dangerously-skip-permissions`,
configurable stage skips, headless timeout doc); (3) sustained
documentation work (methodology.md, SOC 2 walkthrough, FEATURES.md,
audit-driven doc fixes). All additive; no breaking changes to gate
JSON, the adapter contract, or `.devteam/config.yml`.

### Added

- **Test coverage completion: cross-host gate-footer equivalence + verify-stamp middle-path** (audit Batch 2 PR 2.4, addresses P2-6 / T-2 and P2-7 / T-3). Two small additions, both pinning behavior that wasn't being explicitly verified.
  - **Cross-host gate-footer equivalence** in `tests/adapter-contract.test.js`: for a fixed descriptor, claude-code / codex / gemini-cli must render *structurally identical* gate-footer content (normalizing the legitimately host-specific `"host":` literal and the `system_prompt_hash`, since the latter is computed over the full prompt including per-host headers). Pins the `renderStagePrompt` de-duplication (commit 38ce2a0, `core/adapters/render-helpers.js`); silent drift in any one adapter's footer would have caused `devteam reproduce` / `devteam replay` to report hash mismatch across hosts without an obvious culprit. Second test confirms each adapter stamps its own host name. Generic is exempt (different render path).
  - **Verify-stamp middle-path fall-through** in `tests/verify-stamp.test.js`: the command-resolution chain is `.devteam/config.yml pipeline.verify.* → package.json scripts.{lint,test} → skip`. The happy path (config provides) and the skip path (neither provides) were tested; the middle path (no config, package.json scripts present) was unit-tested in `verify-runner.test.js` but not exercised through the full stamping flow. New test: `makeTargetProject` (no verify config) + `package.json` with `scripts.lint: "true"` + `scripts.test: "true"` → both commands resolve, run, and stamp exit_code 0; neither shows up as skipped. Catches any future wiring break between `resolveCommands` and the stamp logic.
  - 783/783 suite (was 780; +3 new tests). 0 lint violations.

- **`docs/user-guide.md` gains an "Ad-hoc Principal rulings" subsection** (audit Batch 2 PR 2.3, addresses D-3 / P2-3). `devteam ruling` was in the README CLI table and the escalation runbook but absent from the user-guide ("daily-use reference" per the README doc map). New subsection right after "`devteam next` says `resolve-escalation`" in "When things go wrong": three concrete "when to reach for it" examples (reviewer `ESCALATE-to-Principal:` marker, ADR-vs-implementation drift, reproducibility-hash drift), the "no gate is written; the ruling is an artifact" framing, all four flags with a real-shape example, routing precedence (`routing.roles.principal` → `routing.default_host`), pointer to the runbook for the full playbook. Doc-only.

- **Operational hygiene cluster: `tryOpen` exec→spawn, `devteam-no-security-review:` magic comment, lazy-require for orchestrator** (audit Batch 2 PR 2.2; addresses P2-2 / S-2, P2-4 / Q-4, P2-5 / P-1).
  - **`core/ui/server.js:tryOpen`** switched from `exec(shellStringWithUrl)` to `spawn(cmd, args, {detached, stdio:"ignore"})`. URL is bound by `server.address()` today (safe), but the `exec(string)` shape is exactly the pattern CodeQL's `js/shell-command-injection-from-environment` rule flags — defense in depth against future URL-source changes. Same fix shape as PR #31's test-plumbing fix.
  - **`core/guards/security-heuristic.js`** gains a `devteam-no-security-review: <reason>` magic-comment override (parity with `secret-scan`'s `devteam-allow-secret:` escape hatch). Content scan returns `[]` when the marker is present, suppressing spurious `stage-04b` triggers on doc/example files that legitimately mention security primitives without using them. 2 new tests in `tests/security-heuristic.test.js`.
  - **`bin/devteam` lazy-loads the orchestrator** via a `getOrchestrator()` helper. The orchestrator is the heaviest module to require (~60 ms cold — transitively pulls in the validator, every gate schema, the spawn helper, and every host adapter via the router). Lightweight subcommands (`help`, `stages`, `hosts`, `doctor`, `init`, `validate`, plus the `memory` / `architecture` / `ci` / `spec` subcommands) no longer pay the cost. Cold-start measured: 40-60 ms → 30-40 ms (~25% on the not-orchestrator path). Hot-path commands (`stage`, `next`, `merge`, `summary`) destructure from `getOrchestrator()` at function entry and pay the load cost only when actually invoked. The lazy pattern is also the right ergonomic shape if the CLI surface grows past 25 subcommands.

- **Lint tooling: `.editorconfig` + ESLint flat config + CI step** (audit Batch 2, P2-1, finding Q-2). Style consistency was being held by author convention through ~9k LOC and 99 commits across 8 days — genuinely consistent, but unenforced. Audit's highest-leverage P2 because the cost of retrofit grows with codebase size. New `.editorconfig` (2-space, LF, UTF-8, final newline, trim trailing whitespace except in Markdown). New `eslint.config.js` (flat config, modern ESLint v9): `eslint:recommended` + `no-unused-vars` (with `^_` ignore pattern for caught errors and intentional args) + `eqeqeq` (strict equality, `null: "ignore"` to allow `== null` for null/undefined checks). Browser globals scoped to `core/ui/static/`; tests/scripts share the Node config. `npm run lint` added; CI runs lint *before* tests (`.github/workflows/test.yml`) so style violations fail fast on Node 20/22/24. Cleanup landed inline: 30 unused-vars violations across `core/`, `bin/`, `hosts/`, `scripts/`, `tests/` (mostly leftover imports from refactors); 3 empty `catch {}` blocks annotated with `/* test cleanup; ignore */`; 1 useless escape in a regex char class. No semantic changes — every fix was deletion of dead code or a small annotation. Added `@eslint/js` + `eslint` as `devDependencies`. Test suite still 780/780 + lint 0 violations. Doesn't introduce `eslint-plugin-security` for now — CodeQL is already catching the shell-injection-shape class on PR scans (caught my own bad test code in PR #31 yesterday). If we want local pre-push enforcement for that class, add the plugin in a future P3 follow-up.

- **`secret-scan` hook gains a 1 MB content size cap** (audit Batch 1, P1-1). `core/hooks/secret-scan.js` previously had no size limit on `tool_input.content`, while sibling components (`core/guards/security-heuristic.js`, `core/hooks/approval-derivation.js`, `core/gates/validator.js`) all cap at 1 MB. Risk was fail-OPEN under load: a model writing a very large file could cause the PreToolUse hook to take long enough that Claude Code times it out, and timed-out hooks are non-blocking — so the secret scan would silently be skipped without any signal. New `MAX_SCAN_BYTES = 1_000_000` matches the sibling cap; oversize content exits 0 with a `[secret-scan] ⚠️ content for <file> is N bytes (cap 1000000); skipping scan and allowing.` warning to stderr. Real source files are well under 1 MB; the cap only fires on pathological cases. 2 new tests in `tests/secret-scan.test.js`: oversize content fails-soft with the warning and the buried-AWS-key is NOT detected (proving the scan was skipped); under-cap content still scans normally. The oversize test uses shell-redirected stdin from a temp file because `spawnSync`'s `input` option EPIPEs on ≳500KB stdin — real Claude Code uses streamed pipes that don't hit this artifact. Closes audit finding S-1.

- **Audit-archive convention + skill step 0.0.** Past audits are now preserved under `docs/audit-archive/<date>-<version>-<context>/` rather than overwritten when a new audit runs. New `skills/audit/SKILL.md` § Phase 0 step 0.0 codifies the archive procedure: check whether `docs/audit/status.json` reports a completed prior audit; if so, `git mv` the prior files into a date-stamped archive directory and append a row to `docs/audit-archive/HISTORY.md`. Skips on `--resume`. Both `/audit` and `/audit-quick` slash commands updated to invoke step 0.0 at startup. New Phase 3.1 wording instructs the auditor to read the most recent archived `09-backlog.md` and `10-roadmap.md` and carry forward any items still open — the mechanism that keeps audits cumulative rather than re-discovering the same gaps each cycle. The existing 2026-05-28 self-audit was moved into the archive (`docs/audit-archive/2026-05-28-v0.4.0-initial-dogfood/`) as the first historical entry; HISTORY.md captures its headline (0 P0 · 5 P1 quick wins · 5 P2 targeted · 1 P3 strategic). New `docs/audit/README.md` doubles as the placeholder that keeps the empty current-audit directory tracked and as discovery for the archive convention.

- **`devteam ruling --topic "..." [--headless]` — ad-hoc Principal subagent dispatch.** Tier-4 closes the "no clean headless path for ad-hoc Principal rulings" gap from the user-report follow-up. Use when an escalation needs a Principal judgment (e.g. a reviewer's `ESCALATE to Principal:`) but you don't want to re-run an entire stage. Dispatches the Principal subagent via the configured host's headless command, with a focused prompt that instructs the Principal to read the cited context (`--context paths` + `--target-gate <gate>`), weigh the question, and write a `PRINCIPAL-RULING:` line into `pipeline/context.md` under a `## Principal Rulings` section. Notably: the ruling is an artifact, not a status change — the user still uses `devteam restart <stage>` (must-fix) or hand-edits the escalating gate (defer) to act on it, so the rule that gates are written by their stage's own dispatch is preserved. Routes via `routing.roles.principal` if set, else `routing.default_host`; refuses cleanly when the host doesn't support `--headless`. User-driven mode (no `--headless`) prints the prompt for paste-into-host. 12 new tests across argument handling, user-driven and headless modes, routing precedence, and host-loading errors.

- **Auto-inject red-team blockers into build context + `--patch` for scoped re-runs.** When stage-04c (red-team) writes `must_address_before_peer_review[]`, the validator now auto-injects each blocker into `pipeline/context.md` so the next build invocation sees the findings without manual copy-paste. New `--patch --from <gate>` flag on `devteam stage build` renders a scoped prompt addressing only the cited blockers, rather than a full re-build. Closes the only place in the pipeline where adversarial feedback was previously broken (red-team would find an issue and the implementer would have to manually port it into context). `core/hooks/approval-derivation.js` writes the injection idempotently. See `tests/red-team.test.js` for the round-trip.

- **QA-within-build fix workflow.** When stage-06 (QA) fails on a fix-the-tests-not-the-implementation case, the orchestrator can now dispatch a scoped build re-run that addresses the QA findings without re-running the entire build stage. Reduces wall-clock on the common "test added late, implementation needs a 3-line tweak" loop.

- **`pipeline.skip_stages` config.** Teams that intentionally skip a stage (e.g., `observability-gate` on a docs-only repo) can now declare that in `.devteam/config.yml` and have `devteam next` walk past the stage cleanly, rather than escalating. Pairs with `core/pipeline/stages.js` so skips are visible at orchestrator level, not buried in role briefs.

- **`nano` track gains scoped peer-review.** Audit Tier-2 policy decision: nano was previously `["build", "qa"]`, which skipped the entire review methodology — "even trivial changes deserve a second pair of eyes." Nano is now `["build", "peer-review", "qa"]` with peer-review configured for one reviewer (the backend slot) and `required_approvals: 1`. New `PEER_REVIEW_SIZING` table in `core/pipeline/stages.js` keys per-track sizing; new exports `rolesForStage(stageDef, track)` and `requiredApprovalsFor(stageDef, track)` that the orchestrator's `computeDispatchPlan` and the `approval-derivation` hook consult. Approval-derivation now reads the project's track from config and stamps the right `required_approvals` (1 for nano, 2 for everything else) and `review_shape` (`single` for nano, `matrix` otherwise) on gate creation. Lighter tracks keep their wall-clock advantage; the marquee review property is preserved across every track. Three new tests across `tests/tracks.test.js`, `tests/next.test.js`, `tests/approval-derivation.test.js`.

- **Stage 7 auto-fold actually implemented.** Audit Tier-2 policy decision: EXAMPLE.md and `docs/concepts.md` documented auto-fold as a shipped feature, but `grep -rn auto.fold core/ bin/` returned zero hits — the orchestrator wasn't writing stage-07.json from stage-06 in any code path. Now it is. When the orchestrator's walk reaches sign-off, it checks: stage-06 is PASS, stage-06's `criterion_to_test_mapping_is_one_to_one` is true, `pipeline/brief.md` has at least one AC-N entry, every AC-N is mentioned in `pipeline/test-report.md`, and no stage-07 workstreams have already started. When all conditions hold, the orchestrator writes `pipeline/gates/stage-07.json` with `auto_from_stage_06: true`, `pm_signoff: true`, `deploy_requested: true`, `runbook_referenced` populated from actual file existence, plus an `auto_fold` audit block listing the criteria it found and stamped. The check is *verified*, not trusted: the orchestrator re-derives the AC list from brief.md and the test mapping from test-report.md itself rather than rubber-stamping the QA agent's claim. 9 new tests in `tests/auto-fold.test.js` covering both fold and refuse paths.

- **Orchestrator-stamped verification gates for stage-04a and stage-06.** Audit Tier-2 finding #1 (CRITICAL): every Stage 04a gate field (`lint_passed`, `tests_passed`) and Stage 06 field (`all_acceptance_criteria_met`) was self-reported by the model that did the work — the validator only enforced shape, not truth. The orchestrator now runs the configured commands itself and stamps what it actually observed. New `devteam verify <stage-id>` command does this on demand; `--headless` mode runs it automatically after the stage completes. When the orchestrator's truth disagrees with the model's claim (e.g., model wrote `lint_passed: true` but the lint command exits 1), the gate's status flips to FAIL and a structured `_orchestrator_stamped` block records the override (commands run, exit codes, model claim, orchestrator observation). For stage-06, the orchestrator also cross-references `pipeline/brief.md` AC-N entries against the `AC | Test` mapping in `pipeline/test-report.md` and overrides `all_acceptance_criteria_met` based on its own count. New `pipeline.verify.{lint,test}_command` config keys; falls back to `package.json` scripts; falls back to skip. Two new modules (`core/verify/runner.js`, `core/verify/stamp.js`) and 27 new tests. Role briefs for `platform` (stage-04a) and `qa` (stage-06) now document the stamping behavior so the model knows its claims will be cross-checked.

- **SOC 2 evidence collector walkthrough.** A non-trivial 17-stage trace at `docs/walkthroughs/soc2-evidence-collector.md` — building a SOC 2 evidence collector CLI end to end, with conditional security/migration stages firing for real, adversarial peer review, and property-based verification. Replaces EXAMPLE.md as the canonical full-feature showcase for evaluators who want to see what the full track actually produces. Linked from user-guide and adoption-guide.

- **`docs/FEATURES.md`** — every shipped feature organized by area. Single source of truth for "what does Stagecraft do?", separate from the methodology doc ("why does it do it that way?") and concepts ("what are the primitives?"). Front-of-line for skeptical evaluators.

- **`docs/methodology.md`** — the development methodology Stagecraft enforces: ATDD loop, phase-gate progression, the adversarial red-team layer, multi-role peer review, and the four coding principles. Pulls together what was previously scattered across coding-principles.md, role briefs, and the rules docs into one referenceable explanation of *why* the pipeline is shaped this way.

- **Red-team failure recovery Q&A in FAQ + unattended-pipeline guidance in user-guide.** Two operational gaps users hit in pilot.

- **Forcing function in dev role briefs: `## Verify` section required before PASS.** Audit Tier-2: dev role briefs (backend, frontend, platform) ended with "Write the gate with `status: PASS`" — no forcing function preventing a careless model from rubber-stamping without doing the work. Each brief now requires the implementing dev to write a `## Verify` section in their `pipeline/pr-{area}.md` listing one bullet per acceptance criterion, each carrying (a) the exact command/action and (b) the observed output. Examples per role: backend bullets show `curl` invocations + actual HTTP responses; frontend bullets show user actions + DOM/screenshot evidence; platform bullets show `docker compose` output + health-check status. The gate-writing step is now explicit that PASS without a populated `## Verify` is invalid and will be flagged at peer review. Pairs with the orchestrator-stamped verification PR's stage-06 AC-mapping cross-check — the model is now both forced to claim verification (in writing) and cross-examined by the orchestrator after the fact.

- **Security heuristic gains content scanning.** Audit Tier-3: the prior heuristic was 12 path regexes — fired on every `package.json` change but missed `src/users/login.ts`, `src/handlers/identity.ts`, and code that imports `bcrypt`/`jsonwebtoken`/`crypto.createCipheriv` without a security-flavored filename. The heuristic now scans both paths and file contents: 10 content patterns covering password hashing (`bcrypt`/`argon2`/`scrypt`/`pbkdf2`), JWT operations, OAuth libraries, Node `crypto` primitives, algorithm constants (`AES-256`, `RSA-2048`, `SHA-512`), session/cookie signing, CSRF, authorization function calls (`hasPermission()`, `requireAuth()` etc.), hard-coded credential literals, SQL string concatenation, and security-context `Math.random` use. New path patterns for `login`/`logout`/`signup`/`oauth`/`identity`/`permission`/`session` directories. New structured `analyze(paths, opts)` API returns findings tagged by kind (path vs content) and pattern label so reviewers see *why* each file flagged. 1 MB size cap on content scanning matches `secret-scan`/`gate-validator`. Back-compat preserved: `needsSecurityReview(paths)` and the CLI exit-code contract unchanged. 16 new tests; the heuristic suite goes from 8 → 24.

- **`adapter-contract.test.js` upgraded from existence checks to behavioural assertions.** Audit Tier-3: the prior contract test was 24 assertions of the shape `typeof adapter[m] === "function"`. A method could `return undefined` and still pass; a regression in install/status/uninstall would slip through to user-install time. Now every adapter is exercised against the contract: `renderStagePrompt` must return a non-empty string containing the workstreamId; `install` must return `{written, skipped}` with the documented shape; `status` must return `{ok, missing}` and report `ok:false` on a clean dir; install+status round-trips to `ok:true`; `uninstall` actually deletes the payload from disk (not just unreported by status); install is idempotent. Plus an `enforces.{allowed_writes,stoplist}` enum check. 32 new assertions across 4 adapters (claude-code, codex, gemini-cli, generic); contract suite goes 24 → 56 tests.

- **Concrete examples inlined into `code-conventions`, `security-checklist`, `api-conventions` skills.** Audit Tier-3: these three skills shipped as bullet-list aphorisms ("Don't use `any`", "No SQL string concatenation", "Use ISO 8601 timestamps") with no examples. Aphorisms are forgettable; a model that's never seen what `any → unknown` looks like in a real diff can't recognise the violation in a review. Each rule now has a paired example showing what BAD and GOOD look like — TypeScript `any` vs `unknown`, parameterised SQL, JWT verify vs decode, 404 vs 403 to avoid existence leaks, timing-safe equality, etc. Rules also gain stable `#rule-name` anchors (e.g. `code-conventions#no-stack-leak`) so reviewers can cite the specific failure mode in `REVIEW: CHANGES REQUESTED` comments rather than restating the rule. Skills go from 55/49/59 lines to 143/144/165; the doubling pays for itself the first time a reviewer can write `BLOCKER: code-conventions#no-any` instead of re-explaining it.

- **De-duplicated `renderStagePrompt` across the three host adapters.** Audit Tier-3: claude-code, codex, and gemini-cli each carried ~30 lines of identical text — gate JSON skeleton, orchestrator/host attribution line, cost-telemetry hint, C4 system-prompt-hash computation + line. Three forks meant every prompt-format change had to be made three times and stay byte-aligned across them or replay hashes would drift. New `core/adapters/render-helpers.js` exports `appendGateFooter(lines, descriptor, ctx, hostName)` and `allowedWritesCaption(enforcementLevel, hostDisplayName)`. Adapters now push their host-specific header (claude-code's subagent + patch-mode block, codex/gemini's role-prompt-path line), the objective/readFirst/artifact sections, then delegate the footer. Net adapter LoC: 780 → 714 (-66 lines of duplication); 69 lines in the helper plus a 10-test unit-test file pinning the hash-determinism contract. Adding a fourth adapter is now a ~5-line footer integration instead of a ~30-line copy-paste.

- **`devteam derive-approvals [<file>]` + Stage-5 quorum-miss documentation.** Two coupled gaps surfaced from a real operator run: (a) Stage 5 can FAIL in two distinct shapes — reviewer objections (existing Case 4 in `fix-and-retry.md`, BLOCKER content present) or a quorum miss (no objections but a per-area gate's `approvals.length < required_approvals`); the runbook only covered the first. (b) The fix we initially documented — append `## Review of <area>` with `REVIEW: APPROVED` to a reviewer file — assumed the `approval-derivation` hook fires on the save, but it's registered as a Claude Code `PostToolUse Write|Edit` hook and only fires on agent-driven writes inside an active host session; shell or editor saves bypass it entirely. New `devteam derive-approvals [<file>]` subcommand exposes the hook to operators with a friendly interface: invokes the same `core/hooks/approval-derivation.js` with a synthetic PostToolUse payload (same code, same gate shape, same lock + atomic-write semantics), defaults to processing every `by-*.md` under `pipeline/code-review/` when no argument is given, validates the file lives under `pipeline/code-review/` and matches `by-<reviewer>.md`. 9 new tests covering end-to-end gate updates (single file, no-arg fanout, quorum flip to PASS), JSON output, and error paths. Docs: `fix-and-retry.md` gains Case 5 (~140 lines) with the worked example and an explicit "Why step 2 is needed: the host-lifecycle constraint" subsection; `concepts.md` gains a 4-row Stage-4-vs-Stage-5 vocabulary callout (`workstreams[]` means implementers at Stage 4 but areas-of-code at Stage 5) and updates the Approval-derivation entry; FAQ entry points at both. The hand-edit gotcha is extended to cover the Stage-5 exception.

- **Per-stage rules files split out of `pipeline-build.md`.** Audit Tier-3: `pipeline-build.md` was 358 lines describing nine stages (4 / 4a / 4b / 5 / 6 / 6b / 6c / 7 / 8). The orchestrator agent loaded the whole file at startup regardless of which stages a given run would actually visit. Each stage's procedural rules now live in its own `rules/stage-NN.md` file (9 new files, 14–144 lines each — stage-05 is the largest because peer review has the most procedure). `pipeline-build.md` becomes a 30-line index that lists every stage with a one-line summary and links to its file. `rules/orchestrator.md` startup instructions updated: read the index, load each `stage-NN.md` on demand as the stage dispatches. `pipeline.md`, `pipeline-core.md`, `pipeline-tracks.md`, and `CONTRIBUTING.md` cross-references updated. Install adds 8 new files (40 → 48 in a fresh `devteam init`). No content changes — the split is purely structural; gate schemas in `gates.md` are unchanged because they're consulted by reference rather than read eagerly.

### Changed

- **Parallel workstream dispatch.** Stage 04 and stage 05 (the two multi-role stages) now dispatch their workstreams in parallel rather than sequentially. On a `full` track with 4 build workstreams, this typically cuts wall-clock by 40–60% depending on host responsiveness. Removed an internal `Promise.all`-shaped redundancy that was already serializing work that didn't need to be serialized.

- **Reduced prompt token usage in claude-code adapter.** Trimmed boilerplate and de-duplicated context-loading instructions in the rendered stage prompt; typical reduction is 15–25% input tokens per dispatch with no observable change in output quality. Direct cost reduction for users on per-token-priced model tiers.

- **`devteam init` and the SOC 2 walkthrough sweep.** Comprehensive doc audit pass over README, user-guide, FAQ, presentation-notes — primarily updating stale stage counts (8→12 roles, 6→13 skills) and aligning the 17-stage table across every reference point. Audit-tier-1 fixes from the 2026-06-02 self-audit completed this sweep (examples, walkthrough, api-conventions DELETE claim, FEATURES.md track counts).

- **Stage 5 renamed from "adversarial peer review" to "peer review" / "multi-model peer review".** The audit flagged that all four area reviewers in Stage 5 use the same `reviewer.md` rubric on the same inputs — a fanout across hosts gives execution-diversity (different training data) but not method-diversity. Calling this "adversarial" overclaimed: the adversarial label belongs to stage-04c red-team (a separate role with a separate method: the 10-surfaces walkthrough). Renamed the term across README, methodology, concepts, FEATURES, user-guide, adoption-guide, presentation-notes, faq, BACKLOG, and EXAMPLE. Stage 4c remains "adversarial review" everywhere — that label is accurate and load-bearing.

- **End-to-end orchestrator state-machine test.** Audit Tier-2: every test in the suite seeded gates directly; no test walked `init → next → stage → write gate → next → ...` against a real tempdir, so a regression in `next()`'s stage-transition logic would be invisible to the existing unit suite. New `tests/pipeline-e2e.test.js` walks both the smallest track (nano: build → peer-review → qa) and the full track (all 17 stages, including the multi-role merge for build/peer-review and the conditional-stage skips for security-review/migration-safety). The full-track walk also exercises Stage 7 auto-fold when that feature is present on the branch. Plus one CLI-subprocess smoke test confirming `devteam init` lays down the expected files and `devteam next` reports the first stage. 4 new tests.

### Fixed

- **Validator auto-injects `orchestrator` and `host` fields.** Gates written by adapters without these fields are completed at validation time using the calling adapter's identity, rather than failing the schema. Catches the most common shape error agents make (forgetting one of the two identity fields) without forcing every adapter prompt to re-state them. `tests/gate-validator.test.js` updated.

- **`--dangerously-skip-permissions` in claude-code headless mode.** Without this flag, headless `claude --print` would hang on the first tool that asked for permission, breaking unattended runs. Set on the headless path in `hosts/claude-code/capabilities.json`; documented as a deliberate trust-the-prompt-discipline tradeoff for the headless mode only.

- **Missing YAML frontmatter on 6 skill files.** Claude Code's skill loader skips skills without frontmatter; six SKILL.md files that shipped during 0.4.0 had been silently invisible. Added the standard `name:` + `description:` block.

- **Malformed ISO timestamps in `examples/sms-opt-in/pipeline/gates/stage-04.*.json`.** Four workstream gates carried `T15:020:00Z` / `T15:010:00Z` (extra digit in minutes); fixed to `T15:20:00Z` / `T15:10:00Z`. The example users copy to seed their own runs is now valid.

- **`docs/walkthroughs/stage-04-split-host.md` rewrite.** The walkthrough was a pre-0.1 design exercise dressed up as a current trace — referenced commands and paths that no longer exist (`devteam status --advance`, `/devteam:stage`, a non-existent pm skill) and proposed a contract that has since shipped. Rewritten as a post-implementation trace using today's CLI, today's `capabilities.json`, and real gate JSON from `examples/sms-opt-in/`.

- **`skills/api-conventions/SKILL.md` DELETE 204 claim.** The skill stated "DELETE returns 204, not 404" as a flat rule contradicting its own 404 entry; now explicit that this is a deliberate project convention (idempotent DELETEs, retry-friendly) rather than a universal HTTP rule, with reviewer guidance about not blocker-ing standards-conformant third-party code.

## [0.4.0] — 2026-05-28

Eight commits since `v0.3.0`. This release closes out the audit's top-11 recommendations (11/11 done, plus E6 as a bonus) and adds two more pipeline stages, taking the total to 17. The arc: record what AI runs (C4) → replay and diff them (E6) → publish gate state to CI (F4) → catch unsafe migrations (B5) → close the AC→spec→test loop (G2) → verify beyond example tests (G7). All additive — no breaking changes to gate JSON shape, host adapter contract, or `.devteam/config.yml` schema. The pipeline is now what an SOC 2 / EU AI Act reviewer asks for: every gate captures *what* configuration produced it, the runtime can replay any past gate and surface drift, CI surfaces gates as check runs, and three new safety stages (migration / executable-spec / verification-beyond-tests) raise the verification ceiling beyond "tests pass."

| Audit theme | What ships it |
|---|---|

### Added

- **G7 — Verification beyond tests (stage-06d).** New stage between observability-gate (06c) and sign-off (07) that applies systematic verification methods AFTER stage-06 (qa) PASS. New `verifier` role (read-only on production code, writes property tests under `src/tests/property/` and formal specs under `pipeline/formal/`) inventories candidates from the diff and picks the right method per code shape: property-based testing (fast-check / hypothesis / proptest) for pure functions / parsers / transforms; mutation testing (stryker / mutmut / mull) for critical business logic with existing test coverage; formal verification (TLA+ / Alloy / Lean) for concurrent state machines / consistency invariants / safety properties. Skip with reason for glue/UI/CRUD. Gate (`core/gates/schemas/stage-06d.schema.json`) records `methods_attempted[]` (enum constrains to property/mutation/formal plus `attempted_but_blocked:<method>` variants for honest reporting when tooling isn't installed), `methods_skipped[{method, reason}]` (reason required minLength: 1 — schema enforces that "didn't have time" isn't a legitimate skip), `candidates_inventoried`, per-method stats objects (`property_based`: properties_asserted/cases_tried/counterexamples_found/tool; `mutation`: mutants_generated/killed/survived/timed_out/score/threshold/tool/target — the pre-declared `threshold` is audit-grade, prevents goal-post moving; `formal`: property_modeled/spec_path/tool/depth_explored/counterexample_found), `findings_count`, `blocking_findings[]` (each requires method + summary), `non_blocking_findings[]`. A surviving mutant on a critical path, a property counterexample to a stated invariant, or a formal counterexample to a safety property populates `blocking_findings[]` → stage FAIL → implementer addresses → re-run (analogous to red-team's `must_address_before_peer_review[]`). New `skills/verification-beyond-tests/SKILL.md` walks five phases (inventory → pick → apply → triage → report) with a code-shape → method selection table, the full property-shape vocabulary (round-trip / idempotence / commutativity / associativity / monotonicity / invariant preservation / reference-implementation / oracle), concrete example invocations per method, and a documented set of failure modes (properties that just re-state the implementation; 100% mutation scores that signal tautological tests; formal proofs of trivialities; counterexample dismissal). New `templates/verification-report-template.md` carries sections per method + candidate inventory + skipped methods + triage + recommendations + the four-shape approval line. Role brief (`roles/verifier.md`) is explicit about distinct-from boundaries (qa = floor, red-team = pre-build adversarial, verifier = post-tests systematic ceiling), the not-faking-runs contract (tool missing → `attempted_but_blocked` with install hint), and escalation triggers (counterexample to spec rather than implementation; 100% mutation score; formal violation suggesting design assumption was wrong). Auto-picked-up contract assertions in `tests/contract.test.js` covered the new stage→schema mapping, role brief presence, and ordered-vs-keys correspondence without modification. Strategic value: stage-06 verifies the cases the author thought of; stage-06d verifies the cases nobody thought of. The shape of the gate makes it impossible to claim verification was done without saying which methods, what was found, and what was skipped + why — drift between "we have tests" and "we have evidence" gets caught structurally.

- **G2 — Closed-loop AC → exec spec → tests (stage-03b).** New stage between clarification (stage-03) and build (stage-04) that translates each numbered acceptance criterion (`AC-N` in `pipeline/brief.md`) into one Gherkin Scenario in `pipeline/spec.feature`, tagged `@AC-N`. - **B5 — Migration safety stage (stage-04d).** New conditional safety stage between red-team (4c) and peer-review (5). - **E6 — `devteam replay <stage-id>`.** Re-runs a recorded stage against the CURRENT configuration, writes the new gate to `pipeline/gates/replay/<stage-id>.<timestamp>.json` (subdirectory, deliberately outside what the validator scans), and diffs the original vs the replay across status / blockers / cost / tokens / duration / reproducibility fields. `--dry-run` prints the plan + prompt-hash drift check without invoking the host. `--json` for tooling. - **C4 — Reproducible runs (recording half).** Every gate can now carry an audit-complete record of how an AI decision was made: optional fields `model_version`, `temperature`, `seed`, `max_tokens`, `system_prompt_hash`, `tools_hash`. `core/reproducibility.js` ships `sha256`, `hashSystemPrompt` (trailing-whitespace-normalized so the same logical prompt hashes the same across line-ending platforms), `hashTools` (sorted + deduped so tool order doesn't change the hash), `reproducibilityFingerprint`, `compareFingerprints`, and `replayReadiness` (classifies a gate as `full` / `partial` / `incomplete`). Strategic value: the gate JSON is now what SOC 2 / EU AI Act compliance reviews ask for — a complete record of what configuration produced each artifact. 24 new tests in `tests/reproducibility.test.js` covering hash determinism + normalization, fingerprint extraction, drift detection, readiness classification, CLI surface, and schema declarations. See `docs/reproducibility.md` for the honest framing of what "reproducible LLM run" actually means (recording vs. determinism).

### Locked design decisions

11 decisions in `ARCHITECTURE.md` — most consequential:

1. The core never spawns a model. Emits prompts and validates JSON.
2. Gate JSON is the stable seam. Identity fields `stage`, `status`, `orchestrator`, `host`, `workstream` (no more `agent`).
3. Role briefs have one source (`roles/*.md`); adapters render into host-expected paths at install time.
4. **Per-workstream host selection.** A single pipeline can route different roles to different hosts; merged via the gate seam.
5. Multi-host install is the default code path. Single-host is a list of length 1.
6. Capability negotiation per adapter — declares `hooks`, `subagents`, `slashCommands`, `worktrees`, `headless`, plus an `enforces` map.

### Differences from claude-dev-team and codex-dev-team

- No fork to maintain — one core, three host adapters.
- Contract F applied everywhere: `agent` field replaced with `orchestrator` + `host` + `workstream`.
- WARN status added (PASS-with-warnings, non-blocking).
- Generic `conditionalOn` mechanism for conditional stages.
- Per-role `roleWrites` map narrows allowed writes per workstream in multi-role stages (backend's prompt no longer lists `src/frontend/`).
- `stage.subagent` override lets multi-workstream stages dispatch all workstreams to a single named subagent (used by peer-review).
- Multi-host install (`devteam init --host claude-code,codex`) is first-class.
